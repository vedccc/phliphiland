Hospitable API v2 — Endpoints We Need

Base URL: https://public.api.hospitable.com/v2. Authentication: Personal Access Token (PAT) passed as Authorization: Bearer {token}. Generate the PAT at my.hospitable.com → Apps → API access → Access tokens.

Endpoints Used

Webhook — message.created

This is the trigger for the entire system. Configured in Hospitable under Apps → Tools → Webhooks.

  • Webhook URL: The Modal webhook endpoint (e.g. https://your-modal-webhook.modal.run)
  • Webhook type to enable: Messages (the message.created action)
  • Payload structure: { "id": "...", "action": "message.created", "data": { ... }, "created": "...", "version": "1.0" }
  • Important: The webhook fires for both guest messages AND host messages. You must filter: only trigger the AI when the sender is the guest, not when you (the host) send a reply.
  • Message age limit: Hospitable will not send webhooks for messages older than 12 hours.
  • Retry policy: 5 retries with exponential backoff (1s, 5s, 10s, 1hr, 6hr) if your endpoint doesn't return 200.
Key investigation needed: Inspect the message.created webhook payload. Confirm it includes: (1) the reservation UUID, (2) the property UUID or a way to derive it, (3) the message body, (4) the sender type (guest vs host). If it only includes the message body and reservation ID, you'll need to call GET /v2/reservations/{uuid}?include=properties to get the property context, and GET /v2/reservations/{uuid}/messages for conversation history.

Section 05

Trigger.dev — Three Workflows

The system has three Trigger.dev workflows. Each is described with its trigger, every step in order, the API calls it makes, and the Supabase queries involved.

Workflow 1 of 3

Main Agent Workflow

Trigger: HTTP POST from Modal webhook (which receives the message.created webhook from Hospitable).
Purpose: Process every incoming guest message through the AI agent. The main agent is a coordinator that delegates tasks to dedicated sub-workflows. Each sub-workflow is a codified piece of logic within the same Trigger.dev workflow, not a separate workflow. Sub-workflows may contain their own AI steps.

Architecture: Coordinator → Sub-Workflows

Think of this as a single Trigger.dev workflow with one main orchestrator (the Claude agent) that picks from four internal code paths. Each code path is a dedicated function with fixed logic, its own API calls, its own database queries, and in some cases its own separate Claude AI calls. The main agent never does the work itself — it classifies the request, passes parameters to the right sub-workflow, receives the result, and uses it to respond.

Phase 1 — Setup (fixed code, no AI)
1
Receive webhook payload from Modal. Extract: message body, reservation UUID, sender type. If sender is the host (not guest), EXIT immediately — we don't process our own messages.
2
Look up property ID. Call GET /v2/reservations/{uuid}?include=properties on the Hospitable API. Extract the property_uuid from the response. If the webhook payload already includes property data, skip this call.
3
Map Hospitable property to our Supabase property. Query: SELECT * FROM properties WHERE hospitable_property_uuid = '{property_uuid}'. If no match, log error and EXIT — property hasn't been synced yet.
4
COOLDOWN CHECK. Query: SELECT * FROM cooldowns WHERE property_id = '{id}' AND is_active = true AND expires_at > NOW(). If a row is returned, EXIT immediately. The AI is inactive for this property.
5
Load conversation history. Call GET /v2/reservations/{uuid}/messages on the Hospitable API. Returns the full message thread. Format as conversation array.
Phase 2 — The Agent Loop

Architecture: The Coordinator is an Agent, Not a Classifier

The coordinator runs as a proper agentic loop using Claude's tool-use pattern. It calls a tool, the orchestrating code executes the corresponding hard-coded sub-workflow, and the result is sent back to the agent as a tool_result. The agent then sees what the tool did and composes the final conversational reply to the guest. The agent is the one who talks to the guest — not the sub-workflows. Sub-workflows perform actions (DB writes, API calls, separate AI steps) and report back what they did. The agent uses that information to write a natural, contextual reply in the guest's language.

The only exception is Sub-Workflow D (Human Escalation). If D fires — either from the agent directly or from A's if/else branch — the entire workflow terminates. No result is sent back to the agent. No reply is sent to the guest. Hard stop.

6
Start the agent loop. Call Claude Sonnet 4.6 with: conversation history, latest guest message, and tool definitions. The agent analyses the conversation and calls a tool. The orchestrating code executes the sub-workflow, sends the result back as a tool_result, and the agent continues — composing its final reply to the guest using the tool's output.

Tool execution. The orchestrating code parses the tool call, runs the sub-workflow function with the parameters + shared context (property_id, reservation_uuid, etc.), and sends the return value back to Claude as a tool_result. Exception: if the tool is escalate_to_human or if Sub-Workflow A branches to D, the workflow terminates — no tool_result goes back.
8
Agent composes final reply. The agent receives the tool result, and outputs a natural, conversational guest message — in the guest's language, informed by the tool output and conversation history. The orchestrating code sends this via POST /v2/reservations/{uuid}/messages.

Why the agent sends the reply, not the sub-workflows

The agent has the full conversation context and the guest's language. Sub-Workflow A returns a raw KB answer — the agent wraps it in a warm reply. Sub-Workflow B returns "ticket created, urgency: high" — the agent turns that into "I've flagged this as urgent and our maintenance team has been notified." Sub-Workflow C returns "approved, Turno task created" — the agent says "Great news, we've arranged that for you!" The agent is the voice. The sub-workflows are the hands.

Workflow 1 — continued

AI Model, Architecture & Prompt Templates

Architecture — this is the most important thing to understand

The main coordinator is an agent running in a loop, not a single prompt. It has access to tools. Each tool maps to a hard-coded sub-workflow — a dedicated block of fixed code within the same Trigger.dev workflow. These sub-workflows execute API calls, database queries, and where needed, their own separate AI calls. When a sub-workflow finishes, it reports back to the agent with what it did. The agent then uses that information to compose and send the final reply to the guest. The agent is the voice. The sub-workflows are the hands. The only exception: Sub-Workflow D (Human Escalation) is a hard stop — it terminates the workflow without reporting back and without sending any reply.

Model

Claude Sonnet 4.6 (claude-sonnet-4-6) for all AI steps. This applies to the coordinator, the KB answerer, and the urgency assessor.

Prompt Format

All prompts use Markdown with consistent sections: # Role, # Context, # Step by Step, # Output. The coordinator prompt also includes # Language since it determines the reply language for the guest.

{{conversation_history}}

Workflow 2 of 3

Property Sync Workflow

Trigger: Manual — triggered by clicking a "Sync Properties" button on the dashboard. No schedule, no cron.
Purpose: Pull properties from Hospitable and Turno and upsert them into Supabase.

Step-by-Step Execution
1
Dashboard user clicks "Sync Properties." This triggers the Trigger.dev workflow via an HTTP call from the dashboard frontend.
2
Fetch all properties from Hospitable. Call GET /v2/properties?include=listings,details. Paginate through all results. Collect: property UUID, name, and any relevant metadata.
3
For each Hospitable property, upsert into Supabase. Query: SELECT * FROM properties WHERE hospitable_property_uuid = '{uuid}'. If it exists, update the name and metadata. If it doesn't exist, INSERT a new row.
4
Fetch all properties from Turno API. Call GET /properties (exact path TBD once API access granted). Collect: Turno property IDs and names.
5
Match Turno properties to Supabase properties. Match by name or by a manual mapping table. Update the turno_property_id column for each matched property.
6
Log sync results. Record how many properties were added and updated. Workflow exits. Dashboard shows the result.
Turno mapping note: Matching Hospitable properties to Turno properties may require a manual initial mapping (done via the dashboard) since the names might differ between platforms. The sync workflow updates metadata but the initial linkage likely needs human confirmation.
Workflow 3 of 3

Supabase Keep-Alive Workflow

Trigger: Trigger.dev scheduled trigger — runs once daily. Not a cron job; Trigger.dev has built-in scheduled triggers that handle this natively.
Purpose: Prevent Supabase free-tier project from being paused due to inactivity.

Step-by-Step Execution
1
Run a simple read query. SELECT COUNT(*) FROM properties. This generates database activity.
2
Run a simple write + delete. INSERT INTO cooldowns (property_id, activated_at, expires_at, reason, is_active, reservation_uuid) VALUES (uuid_generate_v4(), NOW(), NOW(), 'keep-alive', false, 'keep-alive') then immediately DELETE FROM cooldowns WHERE reason = 'keep-alive'. This simulates write activity without leaving junk data.
3
Log success. Optionally log that the keep-alive ran. Workflow exits.

Why this exists

Supabase's free tier pauses projects after a period of inactivity (typically 7 days with no database requests). This lightweight daily ping prevents that. It costs nothing and takes milliseconds to run.

Section 06

How Everything Connects

Data Flow: Guest Message → AI Response

  • Guest sends message in Airbnb/Booking.com/Vrbo →
  • Hospitable receives it in the unified inbox → fires message.created webhook →
  • Modal receives the webhook POST → forwards payload to Trigger.dev →
  • Trigger.dev runs the Main Agent Workflow →
  • Workflow calls Hospitable API for reservation details + message history →
  • Workflow queries Supabase for cooldown check + KB + allowed extras →
  • Workflow calls Claude Sonnet 4.6 with all context →
  • Claude decides action → tool executed: reply via Hospitable API, or create task in Turno API, or SMS via SMS API, or silent escalation to Supabase →
  • Guest sees the reply in their booking platform.

Data Flow: Property Sync

  • Dashboard user clicks "Sync Properties" → triggers Trigger.dev workflow →
  • Calls Hospitable API GET /v2/properties →
  • Calls Turno API GET /properties →
  • Upserts into Supabase properties table →
  • Dashboard reflects current state.

Data Flow: Dashboard Interaction

  • User logs into dashboard (hosted on Render, auth via Supabase Auth) →
  • All reads/writes go directly to Supabase via the REST API or Supabase JS client →
  • User edits KB entries, views tickets, manages cooldowns, configures SMS recipients →
  • Next time the AI workflow runs, it reads the updated data from Supabase.

Section 08

Build Steps — Phase 0: Setup & Webhook Inspection

Before writing any agent logic, we need to set up the infrastructure and inspect the Hospitable webhook payload. This determines whether the plan needs adjustment.

Step 1 — Set Up Modal Webhook
1a
Create a Modal account and set up a simple HTTP endpoint (a Modal web function). This endpoint will receive POST requests from Hospitable and forward the payload to Trigger.dev.
1b
Deploy the Modal webhook. It should accept POST, log the full request body, and return 200 OK. For now, it doesn't need to do anything else — just receive and log.
1c
Copy the Modal webhook URL. You'll need this for the next step.
Step 2 — Connect Hospitable to Modal
2a
Go to Hospitable: my.hospitable.com → Apps → Tools → Webhooks → + Add new.
2b
Paste the Modal webhook URL. Give it a name like "AI Agent Webhook".
2c
Select webhook type: Messages. This is the message.created event. You can also enable Properties and Reservations for future use.
2d
Click "Test". Hospitable will send your latest message to the webhook URL. Check Modal's logs to confirm it arrived.
Step 3 — Set Up Trigger.dev
3a
Create a Trigger.dev project. Set up the workspace and connect it to your Render deployment (or run locally for testing).
3b
Create a test workflow. This workflow does one thing: receive the webhook payload from Modal, log the entire payload to the console, and exit. No AI, no Supabase, no logic — just log what comes through.
3c
Update Modal to forward to Trigger.dev. Instead of just logging, Modal now POSTs the Hospitable payload to the Trigger.dev workflow endpoint.
Step 4 — Inspect the Webhook Payload
4a
Send a real guest message (or use Hospitable's test button). The message flows: Hospitable → Modal → Trigger.dev → logged to console.
4b
Inspect the payload. Document every field. The critical questions to answer:

What We Need to See in the Payload

  • Does it include the reservation UUID? If yes, we can call GET /v2/reservations/{uuid} for full context. If no, we have a problem.
  • Does it include the property UUID directly? If yes, we can skip the reservation lookup for property mapping. If no, we need the reservation UUID to get it.
  • Does it include the message body? The actual text the guest sent.
  • Does it include the sender type? We need to distinguish guest messages from host messages — otherwise the AI would trigger on its own replies.
  • Does it include conversation history? Unlikely — we'll probably need to call GET /v2/reservations/{uuid}/messages separately.
  • What other fields are there? Guest name, check-in date, platform (Airbnb/Booking/Vrbo)? All useful context.
4c
Share the payload with me. I'll review it and tell you if the plan needs to change based on what's actually in the webhook.
Step 5 — Set Up Remaining Services (in parallel)
5a
Create Supabase project. Set up the database, create the super user (liam@vaantq.de), and run the table creation SQL from Section 04.
5b
Create SMSAPI account. Generate the OAuth token, verify a sender name, send a test SMS to yourself.
5c
Contact Turno for API access. Email them now. This runs in the background while we build everything else.
5d
Get Anthropic API key. Create a key at console.anthropic.com if you don't have one already.
5e
Set all env variables in your Trigger.dev project and Render deployment.

After Phase 0

Once we've inspected the webhook payload and all services are connected, we'll know exactly what data we're working with and can proceed to building the actual agent workflow. If the payload is missing key fields (like property UUID or sender type), we'll adjust the setup phase steps in the main workflow accordingly.
